日常实用小技巧｜02

怎样写提示词？

不是写“咒语”，而是把任务说清楚、管起来、检查好。

把任务写清楚，而非写得神秘

01让任务结构更清楚

Markdown 写法
# 任务
为新品发布写一封内部通知

## 背景
面向销售团队，清晰、专业

## 输出
- 300 字以内
- 包含 3 条行动项
YAML 写法
任务: 为新品发布写内部通知

背景:
  读者: 销售团队
  语气: 清晰、专业

输出:
  字数: 300字以内
  包含: 3条行动项
格式不是门槛；它只是把信息分层写清楚。

02让协作过程更可靠

1

复用 Skill｜把常用方法封装起来

可在 GitHub 等平台寻找；使用前先看来源、内容与适用范围。

2

先做 Plan｜先有大纲，再开始执行

没有内置 Plan？先让 AI 写一份行动指令（Action Directive）。

3

同轮自检｜在输出前回答一组检查问题

自检问题（CoT）示例：有没有偏离核心要求？有没有事实错误或前后矛盾？输出格式是否达标？
4

别事后硬找错｜容易为了迎合而“找出问题”

用明确标准和同轮自检，替代没有标准的事后挑错。

5

让 AI 帮你写｜格式、Plan、自检清单都可先让 AI 起草

把“怎样与 AI 协作”本身，也变成一项可委派的结构化任务。

聪明的用法：让 AI 帮你把“怎样与 AI 协作”也写清楚。

知识库打造：优先整理成 Markdown

把常用资料变成轻量、清晰、可按需引用的 AI reference。

不是“把所有附件都丢进去”

01先把资料换成 AI 容易读的形态

PPTPDF

不适合直接当知识库

版式、图片、分页和重复对象都会增加解析负担。

Token 占用大
结构识别容易漏读
局部引用不灵活
整理转换
.MD

Markdown 更适合 AI

纯文本、标题、列表和字段，层级明确，也更节省上下文。

Token 占用更省
结构识别清楚
局部引用精准
原则：保留原始 PPT / PDF 归档；另整理一份 Markdown 给 AI 使用。

02再把知识库做成“索引 + 小文件”

1

按主题拆成小文件

品牌、产品、规则、案例分别维护，避免一个巨型文档。

2

用 README 做总索引

告诉 AI 每个文件是什么、什么时候应该引用。

3

每次只带必要资料

写某条线路，只引用品牌基础和对应产品文件。

再点击一次：查看真实知识库示例

知识库的价值，不是“资料更多”，而是让 AI 每次只读到真正需要的那几份。

Part 4｜DATA SECURITY

数据安全与公司资产：AI 使用的底线

公司信息不是提示词素材；技术可以提效，授权边界不能被跳过。

全员必守 · 安全底线
默认原则：没有明确授权，不要把非公开信息交给外部 AI。

公司数据、代码、合同、品牌素材和业务文件，都是需要受控管理的公司资产。

先判断 · 再脱敏 · 后上传

绝对不要上传

非公开、敏感、可识别的信息
×未公开战略、经营、财务信息与项目底牌
×客户、员工、合作方的个人信息与业务数据
×源代码、账号、密码、密钥及系统配置
×合同、报价、底价、内部邮件与会议材料

确需使用时，先处理

授权接口 + 最小必要原则
1只接入公司批准的接口，并按流程申请所需权限
2先脱敏、匿名化：名称、金额、项目名用占位符
3只提供完成任务所需的最小片段，不整包上传
4按公司制度确认数据留存、训练与共享设置

使用后仍需负责

AI 不替代人的审核责任
✓输出进入业务前，复核事实、版权与引用来源
✓生成文件继续按权限、版本和归档规则管理
✓不把 AI 输出直接当作审批、法律或合规结论
!发现误传或异常共享，立即停止并按流程上报
上传前先问三句公开了吗？→必须给吗？→接口 / 权限获批了吗？

拿不准，就先不上传：先咨询直属主管、信息安全或法务，再继续使用 AI。
